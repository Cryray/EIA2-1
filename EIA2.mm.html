  • + - Coding Style
    • + - Purpose

      • Programme können schnell sehr komplex werden. Daher ist es wichtig, sich an Stil-Regeln zu halten, um sie möglichst verständlich zu schreiben. In diesem Kurs gelten folgende Stil-Regeln:

    • + - 1. Self-explanatory

      • Code sollte sich so gut wie möglich selbst erklären. Hierzu sind sprechende Variablen-, Funktions- und Klassennamen etc. erforderlich. Kurze Namen sind nur in kleinen Gültigkeitsbereichen oder bei klarer Bedeutung (z.B. y für vertikale Position) erlaubt.

    • + - 2. Explicit typing and semicolons

      • Variablen erhalten explizite Typ-Notationen und Anweisungen werden mit einem Semikolon beendet (auch wenn TypeScript diese automatisch einfügen bzw. inferieren kann)

    • + - 3. camelCase for names of variables and functions, functions describe activities

      • Variablen- und Funktionsnamen beginnen mit Kleinbuchstaben und folgen der Kamelnotation (camelCase), d.h. bei zusammengesetzten Namen beginnen die Wortteile im Inneren mit einem Großbuchstaben  (z.B. animalLion). Funktionsnamen beschreiben dabei eine Aktivität (z.B. calculateHorizontalPosition(...)) oder Frage (z.B. isHit())

    • + - 4. Names of formal parameters start with underscore
      • Die Namen formaler Parameter in Funktionen beginnen mit einem Unterstrich.
    • + - 5. PascalCase for classes, interfaces and modules

      • Die Namen von Klassen, Interfaces und Modulen beginnen mit einem Großbuchstaben und folgen ebenfalls der Kamelnotation. (PascalCase)

    • + - 6. All capital letters for enums, support readability using underscores

      • Die Namen von Enumerations und deren Elemente werden durchgehend mit roßbuchstaben geschrieben. Wortteile werden bei Bedarf mit Unterstrich getrennt.

    • + - 7. Use double quotes for literal strings
      • Literale Zeichenketten werden in doppelte Anführungszeichen geschrieben.
    • + - 8. Use comments for separation and to support clarity. Use JSDoc-Format for reuse in other programs

      • Kommentare werden eingesetzt, um Programmteile abzugrenzen und die Verständlichkeit zu erhöhen. Programmteile, die von anderen Skripts genutzt werden sollen, werden im JSDoc-Format kommentiert
        (/** … */)

  • + - Dictionary
    • + - namespace
      • defines a section of storage
      • + - to store data as variables, functions, arrays, classes, objects etc.
        • basically a sequence of bits
        • to be interpreted in different ways
      • belongs to a function, class-instance, module etc. or is global
      • + - scope
        • the namespaces accessible from a particular part of the program
        • the global namespace is accessible everywhere
      • + - Analogy
        • storage holding lots of different stuff, that gets thrown in
        • not all storages can be accessed from everywhere
    • + - type
      • the meaning of data, the way to interpret a sequence of bits
      • + - Analogy
        • the kind of some stuff
        • eg. solid, liquid, gaseous
    • + - variable
      • small piece of storage in a namespace that holds a value of a given type
      • has a name unique to the namespace, by which the content can be found
      • + - variables of
        • + - simple type
          • are standard containers holding the value itself
        • + - complex type
          • hold a reference to data that is stored somewhere else
      • + - test type
        • + - typeof
          • returns the type of a simple variable
          • symbol, function, object for variables of other types
          • or undefined
        • + - instanceof
          • checks the type of a class instance against a given class
      • + - Analogy
        • container taylored for a specific kind of stuff
        • like cardboard boxes, bottles or balloons
    • + - function
      • piece of code that can be executed arbitrarily within the program
      • comes with its own namespace to temporarily hold local variables
      • + - has a name unique in its scope, and is called to execute by this name
        • exception: anonymous functions that are referenced in a specific context
      • can be defined to accept data as parameters of specific type
      • can be defined to return data of specific type
      • + - Analogy
        • + - machine, that can
          • get stuffed with input of specific kind
          • do stuff, using the input if given
          • create, delete or mainpulate stuff in its scope
          • produce stuff of a given kind as an output
    • + - parameter
      • variable defined in the head of a function, through which data can be passed to the function
      • parameters sit in the functions namespace
      • + - Analogy
        • a slot of the machine, where stuff of a specified kind can be put in ...
        • ... for the machine to work with
    • + - array
      • piece of storage in the namespace with variable size
      • that is addressed with a single name
      • and devided into elements of a given type
      • elements are addressed with a number, starting with 0
      • + - Analogy
        • a drawer cabinet in the storage, that can be found with a single name
        • drawers are numbered in sequence starting with 0
    • + - associative array
      • like an array
      • elements are not adresses using a number, but a key
      • in most cases, the key is of type string
      • this is the fundamental data structure in JavaScript, thus also called javascript-object
      • + - Analogy
        • a drawer cabinet in the storage, that can be found with a single name
        • drawers are labelled
    • + - interface
      • a pattern for an associative array
      • + - for each instance of an array that implements the interface
        • + - homogenous
          • the interface defines a type for the keys and a type for the associated values
          • no limitation to the number of elements
          • all elements are addressed with the given type of key
          • all associated values are of the given type
        • + - heterogenous
          • the interface defines particular keys and a type for each associated value
          • elements limited to the defined keys
      • + - Analogy
        • a construction plan for a drawer cabinet with labels
    • + - class
      • a pattern for a complex data structure ...
      • ... using assigned variables, then called attributes or properties ...
      • ... and functions, then called methods
      • + - Analogy
        • a building plan for a studio
        • with assigned storage and machines
    • + - method
      • a function defined in a class
      • has access to the namespace of the class using the operator "this"
      • + - Analogy
        • a machine in a studio
    • + - object
      • an instance of a class
      • each object has its own namespace...
      • ... and addresses its data using the keyword "this"
      • + - Analogy
        • a studio that was actually built
    • + - constructor
      • a special method, that is automatically called on instantiation of the object
      • identified by the name "constructor", which is a keyword
      • the keyword "new" instantiates an object from a class
      • + - Analogy
        • a machine that helps building a studio from the building plan
    • + - subclass
      • a pattern for an extension of a class
      • can add more attributes and methods ...
      • ... or redefine them
      • keyword: extends
      • + - Analogy
        • extension to an existing building plan for a studio
        • more or different machines and containers
    • + - superclass

      • the class a subclass is an extension of

        an object of the subclass can specifically address its superclass namespace using the keyword "super"

        + - Analogy

        • the building plan, that is the foundation of the extension
    • + - inheritance
      • a subclass inherits all the attributes and methods of the superclass
      • + - Analogy
        • a studio of an architect and a comic artist ...
        • ... may both come with a drawer cabinet for different paper and a printer ...
        • ... which is the standard infrastructure of a general studio for drawing
    • + - polymorphism
      • a variable of the type of a superclass...
      • ... can reference an instance of its subclasses ...
      • ... or their subclasses etc.
      • + - Analogy
        • an architects studio and a studio for the comic artist are still studios
        • they just have different extension
    • + - access modifiers
      • also called visibility modifiers
      • limit the usage of attributes and methods of an object
      • + - private
        • only the object of the particular class may access
      • + - protected
        • additionally, objects of its subclasses may access
      • + - public
        • access is allowed from any part of the program
      • + - Analogy
        • can everybody, who knows the studio, just walk in and use its stuff and machines?
        • can somebody, who sits in the extension of the studio, use the stuff in the main studio?
    • + - static
      • attributes or methods defined as static are defined not in the namespace of an object...
      • ... but in the namespace of a class
      • all objects of the class share these, and they are available even if no object is instantiated
      • + - Analogy
        • the studios of the same kind use some common stuff
        • that stuff sits in a common storage
    • + - module
      • a collection of classes and functions to be used in different programs
      • uses a separate namespace
      • + - Analogy
        • a plan for an agglomeration of studios, machines and stuff of different kinds ...
        • ... and their building plans
    • + - generics
      • code structure capable of working with variable types of data
      • types to work with are defined when using the structure
      • + - Analogy
        • building plan for a studio, which keeps some information open for later specification
        • e.g. the kind of stuff that goes into a cabinet or the kind of stuff a machine produces
    • + - enumeration
      • a set of named numeric constants, referenced by arbitrary names in code
    • + - exception

      • a block of code marked with the keyword "try" ...

        ... will not stop the program when a fatal error occurs ...

        ... but instead skip execution and continue with the following block marked with the keyword "catch".

        if a block marked "finally" is present, it'll be executed before either the try- or the catch-block terminate

        the statement "throw" intentionally causes an exception and passes execution to the logically nearest catch

  • + - Eventsystem
    • + - General
      • eventsystem enables the programm to react to changes in the environment
      • programmer doesn't require detailled knowledge of how to detect those changes
      • when a change happens, an "event gets dispatched"
      • event triggers listener-functions, that were explicitly registered for it
      • + - events like mouseclicks or keypresses travel through the DOM in three phases
        • + - capture
          • gets passed from the window into the branch of the DOM tree ...
        • + - target
          • ... where the node is located the event refers to ...
        • + - bubble
          • then gets passed back until it reaches the window again
      • + - information about the event is passed as parameter to the listener function
        • in case of DOM-Event as Event-Object
        • + - function should accept this as parameter
          • eg. function handler ( event: Event ): void
        • definition for the interface EventListener
    • + - Implementation
      • + - recommended
        • someNode.addEventListener(type: string, listener: EventListener, useCapture?: boolean): void
      • + - acceptable
        • + - someHTMLElement.onclick = listener;
          • there can be only one...
          • no control over phase
      • + - discouraged
        • in HTML-File
    • + - Event-Object
      • + - target
        • + - node the event refers to
          • e.g. the clicked button , the changed inputfield
      • + - currentTarget
        • the node the currently processed listener was added to
      • + - type
        • see DOM-Event-Types
      • Mouseposition relative to several coordinate systems
      • Keyboardcodes
      • Eventphase
      • etc.
    • + - DOM-Event-Types
      • + - Window/Object
        • + - load
          • loading of a file is done
        • + - unload
          • the page gets removed from memory
        • + - error

        • + - abort

      • + - Pointer
        • + - pointerover
          • fired when a pointing device is moved into an element's hit test boundaries.
        • + - pointerenter

          • fired when when a pointing device is moved into the hit test boundaries of an element or one of its descendants, including as a result of a pointerdown event from a device that does not support hover (see pointerdown).

        • + - pointerdown
          • fired when a pointer becomes active.
        • + - pointermove
          • fired when a pointer changes coordinates.
        • + - pointerup
          • fired when a pointer is no longer active.
        • + - pointercancel

          • a browser fires this event if it concludes the pointer will no longer be able to generate events (for example the related device is deactived).

        • + - pointerout

          • fired for several reasons including: pointing device is moved out of the hit test boundaries of an element; firing the pointerup event for a device that does not support hover (see pointerup); after firing the pointercancel event (see pointercancel); when a pen stylus leaves the hover range detectable by the digitizer.

        • + - pointerleave

          • fired when a pointing device is moved out of the hit test boundaries of an element. For pen devices, this event is fired when the stylus leaves the hover range detectable by the digitizer.
